Fast Track Requests

A pilot initiated to complete high priority simple request within a day to ensure that client gets the required information on live.


Current Approach

  1. 1. LAL raises a request with a priority and expected go-live date.
  2. 2. Off shore PM to check the feasibility and completeness of the requirement and the expected date of delivery.
  3. 3. Off shore development team to work on the requirement and send it for Quality Checks.
  4. 4. Moved to LAL’s queue for Approval to publish the request after seeking consensus from the client.
  5. 5. Development team publishes the request.
  6. 6. Once the changes are live, close the request.

Fast Track Approach

  1. 1. LAL raises a request with a priority and expected go-live date.
  2. 2. Off shore PM to check the feasibility and completeness of the requirement and the expected date of delivery.
  3. 3. Off shore development team to work on the requirement and send it for Quality Checks.

  5. 5. Development team publishes the request.
  6. 6. Once the changes are live, close the request.

Assumption with Fast Track Approach

  1. 1. Number of ticket per day (month) on Fast track mode – 2 or 3 (35-40) tickets
  2. 2. LAL's will be accountable for the Quality checks of the requests. IDC QA team will not be available for quality checks
  3. 3. Fast track support will be available between 9:00 CET – 18:00 CET
  4. 4. Turnaround time per ticket – 24 hrs. (end-to-end) i.e. 1 working Day
  5. 5. The type of tickets to be considered under this category are –
     
  6. a. Adding / Modifying / Removing Promotion Prices and text and/or legal text
  7. b. Adding / Removing Images from slider, promotions and/or teaser
  8. c. Adding / Modifying / Removing Pricelist brochures
  9. d. Modifying / Removing Menu links
  10. e. Modifying / Removing Call to Action and/or links
  11. f. Adding / Modifying / Removing SEO tags (not more than 10 URL’s)
  12. g. Adding / Modifying /Removing Analytics tags (not more than 10 URL’s)
  13. h. Adding / Modifying / Removing News Updates

6. The types of tickets NOT considered under FAST TRACK CATEGORY would be as follows –

  • a. New Model Creation
  • b. New Page Creation
  • c. Form field changes / Forms issues
  • d. New form creation
  • e. Tickets Needing changes in JavaScript / CSS
  • f. SEO Redirects
  • g. Adding new functionality and Modifying / Removing existing functionality
  • h. Changing existing functionality of Call to Actions / Links / Images
  • i. Creating / Modifying DEM’s and Landing Pages
  • j. Creating / Modifying Banners from Master Banners
  • k. Any request needing template modification
  • l. Any request having root level changes

  • 7. Tickets raised under FAST TRACK CATERGORY will NOT have any EXTERNAL DEPENDENCY on Client information and/or 3rd Party Dependency (Weedoo, Open Reply etc.). A ticket having any external dependency will be cancelled or moved regular JIRA queue.
  • 8. Any requests without information (Waiting for LAL) for completing the ticket will be cancelled or moved to regular JIRA flow
  • 9. LAL's to ping the dedicated fast track resource to initiate the process.
  • 10. No Change requests shall be requested on fast track requests. A change request will be treated as new request requiring a new ticket
  • 11. Any delays caused due to environment issue such as CMS down will be communicated to the LAL’s and delivery time should be re-considered or the request should be moved to normal JIRA Queue
  • 12. Any requests stuck due to infrastructure issues such as Akamai purging (even after following guidelines) should be picked up by LAL’s and send across to the right teams.
