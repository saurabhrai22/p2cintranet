1.Types of Issues

On the “FCA P2C – Content Adaption” project we have 3 categories of issues as follows –

  1. Task
  2. Fast Track
  3. Query

Task

A requirement from HQ or LC where development efforts are needed would be raised as Task. Tickets raised under this category will be included in the monthly ticket count.

Fast Track

A pilot project has been initiated with few markets, any requirement from HQ or LC that falls under fast track category as per the “Fast Track Request” document can be raised under this category. Tickets raised under this category will be included in the monthly ticket count.

Query

Any request to check the feasibility of the requirement or research for workable solution for the client would fall in this category. Tickets raised under this category will not be included in the monthly ticket count. Any development activity required as a result of the output of Query ticket will not be done on the same ticket. A new ticket under task or fast track should be raised and the ticket rules should be followed.

2. JIRA Workflow

The 3 categories of the tickets that can be raised on “FCA P2C – Content Adaptation” project have their respective workflow as show in each of the sections below.

Definition of JIRA Stages
  • Open : The request received by LC and/or HQ will be raised on JIRA by the LAL’s for IDC team to action upon
  • In Progress : The tickets raised by LAL are checked for feasibility and moved to into IDC Queue for action. The stage of the ticket would be “In Progress”
  • In QC : At this stage the request is being checked against the requirement provided by the LAL’s
  • Approval Pending : The ticket is waiting for LAL quality check and approval
  • Pending for Client : Ticket is waiting for approval from Client
  • Ready to Publish : The ticket is approved by all stakeholder and can be published on the Live site
  • External Technical : A ticket is moved to External Technical (ET) when IDC sees technical Limitations in completing the development activities (support from Fiat Support is needed)
  • Waiting for LAL : When IDC needs clarification on the requirement mentioned in the ticket, the ticket is moved to Waiting for LAL
  • External Business : When a ticket needs inputs from HQ/LC, the ticket is moved into External Business (EB) by the LAL’s
  • On Hold : When LC/HQ decided to postpone the development of the request due to incomplete assets or delay by business, the request can be put “On Hold”
  • Cancelled ACN : This is used when ACN did not work on the given requirement. These tickets will not be included in the monthly ticket count.
  • Cancelled LC : Cancelled LC – This is used when ACN has worked and LC/HQ decided to cancel it after the work has been done. These tickets will be included in the monthly ticket count.

3. JIRA Tickets Management

Based on the category of under which the ticket is raised some fields would be mandatory. All mandatory fields would be highlighted with <<*>> are Mandatory.

JIRA Field Details
  • REPORTER will always be the stakeholder creating the ticket
  • ASSIGNEE is pre-set on «Automatic»
    • If you need to assign the ticket to someone specific, simply key in the name and select it from the drop-down menu that will appear
    • If your ticket is related to a Bug Tracker issue, you should use the «Assign to me» option
  • MODE is pre-set on «None»: always be sure to PRODUCTION mode. TRAINING or REHEARSAL modes might apply in specific cases and will not count in the reporting.
  • COUNTRY/LANGUAGE
    • For multilingual countries, such as Belgium and Switzerland markets always create 1 ticket for each language using CLONE option
  • BRAND is pre-set on «None»: always be sure to select the right Brand
  • PLATFORM is pre-set on «None»: always be sure to select the right one
  • CMS platform use DESKTOP & MOBILE option only for responsive websites (Abarth, Fiat, Alfa Romeo), otherwise always create 2 tickets (one for Desktop and one for Mobile platform)
  • REQUEST START DATE is the date requested to start working the ticket
  • EXPECTED DELIVERY DATE BY IDC is the date requested for IDC to complete development and Quality Check
  • EXPECTED GO LIVE DATE is the date requested for the GO LIVE
  • REQUEST PRIORITY is pre-set on «None»: always be sure to select the right one between LOW, MEDIUM or HIGH
  • SUMMARY to describe in a few words the subject of the ticket
  • DESCRIPTION to fully describe and add specification about the ticket
  • ATTACHMENT browse to attach files to your ticket
  • NOTES to add notes
  • ANALYTICS TAG – to define if is tagging is required
  • DELIVERY TEAM – to segregate the tickets done by IDC and fiat support. This is used to avoid double billing by IDC and Fiat support.
  • TOTAL NUMBER OF REWORKED PRODUCTION TASK – to keep track of number of activities requested by the client from the start to the end of the cycle.

Category Details

REQUEST TYPE is pre-set on «None»: always be sure to select the right one between the following
  • Banner for format adaptions and adjustment (copy or image) on existing format
  • CE Activities if JS or CSS adaptations are needed, for image and copy adaptation, for new pages within the CMS, for bug or adaptation not in our scope (i.e. car configurator)
  • DEM for adaptation of non-Eloqua DEM, for adaptation on HQ DEM to accomplish Market needs
  • LP for adaptation on existing Landing Pages, for adaptation on HQ Landing Pages to meet LC needs
  • HQ Model Launches new models being launched. This category is reserved to specific launches agreed with HQ. Please do not use this category unless specific instructions are provided.

  • Banner ad. Including Flash to HTML conversion – Flash banner needs to be converted to HTML5 banner

  • Call to Action/Links Addition / Removal / Changes of CTAs & CTA url managed through coding (not embedded in image); clickable text and corresponding url for the text
  • Broken Links – Changing URL on the CTA or removal of the CTA ensuring that the user is not directed to 404 page. This sub category will only be used for the tickets raised because of broken link report sent by IDC. Tickets raised under this category will not be included in the monthly ticket count.
  • Image Modification – Addition / Removal / replacement of images on the site

  • JavaScript/CSS Updates/Issues/Changes – Any JavaScript / CSS changes, additions, deletions of the pages wherever possible in IDC access /scope control
  • Navigations/Menu Updates/Issues/Changes – This is the same category as Header/Footer/Menu changes and Call to Action Links. Please check if those categories can be utilized
  • New Forms – Creating new forms for new models. This is non-Eloqua forms
  • New Models Creation - New models / New page creations / Activities required for adding CPS on sites
  • News Updates - Creation of News pages or removal / modification of existing news pages and corresponding content in the Hub pages and home pages’ wherever applicable
  • Promotions Non-CPS - Any promotion addition / modification / deletion that are not on CPS platforms
  • Promotions CPS - Any promotion addition / modification / deletion that are on the CPS platforms

Ticket Rules
  • 1 ticket needs to be created for each Brand/Market/Language/Platform. The number of tickets for monolingual, bilingual and Trilingual countries would be as shown in the image below.
  • For all responsive brands, such Fiat, Alfa Romeo and Abarth, there should be only 1 ticket for each Brand/market/Language/Platform and the platform should be “Desktop & Mobile”.

Ticket rules for each category/sub category of request are as follows –
  • Banner Category - Each banner counts as 1 ticket. This ticket will include the creation of banner (flash/HTML) + fallback images (static). Additional sub task will not be created for fall back images. All requests raised under banner category Local account leads would create 1 main task for Banner adaptation request for each Brand/market/Language/Platform. IDC team would be responsible for the calculation of banner format required and for each banner format 1 sub task would be created.
    The sub categories that follow this rule are - Banners - Adaptation from Master, Dynamic Banner - first set up of a campaign, Banners - Minor Changes/Modification and Banners - Resizing/Reformatting
    For Ex – If a request is raised for BE-FR to adapt banner having 3 formats mainly 300x600, 970x250 and 160x600, then LAL’s are required to raise 1 ticket and IDC would create 2 sub tasks to this sub category.
  • Analytics Sub category – LAL’s are requested to raise 1 ticket for each Brand/market/Platform for up to 50 URL’s. IDC would go ahead and create 1 sub task from 51st URL for the next 50 set of URL’s.
  • Analytics Tags Bugs – This category would be only used by the analytics team to report the bugs generated during analytics testing. This sub category should not be used LAL. Tickets raised under this sub category will be excluded from the monthly ticket count.
  • Call to Action/Broken Links Sub category – All LAL’s are requested to raise 1 ticket for each Brand/market/language/Platform up to 10 URL’s. IDC would go ahead and create 1 sub task from 11th URL for the next 50 set of tickets.
  • Feature/Functionality – 1 ticket for the implementation of a new functionality which has not been implemented previously or not present on any other websites.
  • Images – For this sub category, when the graphics team is involved 6 images would be accommodates per ticket LAL’s are requested to raise 1 ticket for up to 6 images. IDC would go ahead and create 1 sub task from 6th image for the next 6 set of images. If the only activity is to upload the images given by HQ/LC on the CMS where graphics team is not involved 25 images would be accommodated in 1 ticket.
    For Ex – If there is request raised by LAL to work on images for sliders which require Graphics Producers to work and finalize the images, IDC would go ahead and create a sub task for Graphics team to work on. Each of these sub task would accommodate 6 images, once the images are ready content team would go ahead and upload the image in CMS. On the other hand, there are times when HQ/LC has given the final images and content team is required to upload the images in CMS. Under such circumstances, 25 images would be accommodated in 1 ticket
  • Multiple Content – For requests under this sub category LAL are required to create 1 main ticket for each Brand/market/language/Platform and requirements should be attached as PPT. IDC would go ahead and check the number of slides per PPT and accommodate up to 25 changes on each ticket. From the 26st change, IDC would go ahead and create a sub task for the next 25 set of changes.

    NOTE: A change can be changing 1 whole paragraph or changing 1 line or changing 1 single word or changing 1 single character (e.g. removing a semi colon)
    For Ex – If a monolingual market attaches a PPT having 65 changes in a single ticket, IDC would go ahead and create 3 subtasks. First 25 changes would be marked to the main ticket. For the next 25 changes IDC, would create 1 subtask, following 15 changes would be worked on in the 2nd sub task. If within these slides, graphical changes are required and IDC needs to involve the graphics team, 1 sub task would be created for graphics activity. The slides requiring graphical changes would be excluded from the content count and included in graphics counts. For Graphical work, 1 ticket would accommodate 6 image changes.

    For Bilingual and trilingual markets, LAL’s should create a ticket for every language within the market. The number of slides on slides for a language would be calculate and the same logic would follow.

    For Ex – If a trilingual market attaches a single PPT have 65 changes in a single ticket. IDC checks the number of slider corresponding to the 3 languages of the market. Language 1 has 20 changes, language 2 has 20 changes and language 3 had 25 changes, then the ticket below to the 3rd language would have 1 subtask. The tickets for language 1 and language 2 would not have any subtasks. If within these slides, graphical changes are required and IDC needs to involve the graphics team, 1 sub task would be created for graphics activity. The slides requiring graphical changes would be excluded from the count and included in graphics counts. For Graphical work, 1 ticket would accommodate 6 image changes.
  • SEO Redirects – This sub category is in IDC control only for JEEP Desktop and partially for Mobile. LAL’s would need to create a ticket for each Brand/market/Platform. First 50 URL’s would be accommodated on the main task. For the 51st URL, IDC would go ahead and create 1 sub task for the next 50 set of URL’s.
  • SEO Tags - LAL would need to create a ticket for each Brand/market/Platform. First 50 URL’s would be accommodated on the main task. For the 51st URL, IDC would go ahead and create 1 sub task for the next 50 set of URL’s.
  • Text Modification – Check if this can be removed from JIRA
  • Adding Video - LAL would need to create a ticket for each Brand/market/Language/Platform. First 10 videos would be added, updated or removed on the main task. For the 11th video, IDC would go ahead and create 1 sub task for the next set of 10 videos.
  • Bug/Technical Issues –LAL would need to create one for ticket for each page. If 3 bugs are spotted on a page those 3 bugs can form 1 ticket. Please note, that the tickets created under this sub category are excluded from the monthly ticket count.
  • Form Field Changes – LAL would need to create 1 ticket for each form. This category would only comprise for changes (not bugs) of Non-Eloqua forms.
  • Form Privacy Changes - LAL would need to create 1 ticket for each form. This category would only comprise for changes (not bugs) of Non-Eloqua forms.
  • Header/Footer/Menu Changes – For requests under this sub category LAL are required to create 1 main ticket for each Brand/market/language/Platform and requirements should be attached as PPT. IDC would go ahead and check the number of slides per PPT and accommodate up to 25 slides on each ticket. From the 26st slide, IDC would go ahead and create a sub task for the next 20 set of slides. Please also note, that each slide of a PPT document can have only 1-2 low complexity change.
  • HTML Updates/Issues/Changes – For requests under this sub category LAL are required to create 1 main ticket for each Brand/market/language/Platform and requirements should be attached as PPT. IDC would go ahead and check the number of slides per PPT. IDC would go ahead and check the number of slides per PPT and accommodate up to 2 slides on each ticket.
  • JavaScript/CSS Updates/Issues/Changes – For requests under this sub category LAL are required to create 1 main ticket for each Brand/market/language/Platform and requirements should be attached as PPT. IDC would go ahead and check the number of slides per PPT. IDC would go ahead and check the number of slides per PPT and accommodate up to 2 slides on each ticket.
  • Navigations/Menu Updates/Issues/Changes – For requests under this sub category LAL are required to create 1 main ticket for each Brand/market/language/Platform and requirements should be attached as PPT. IDC would go ahead and check the number of slides per PPT and accommodate up to 25 slides on each ticket. From the 26st slide, IDC would go ahead and create a sub task for the next 20 set of slides. Please also note, that each slide of a PPT document can have only 1-2 low complexity change.
  • New Forms Updates/Issues/Changes – LAL would need to create 1 ticket for each form. This category would only comprise of Non-Eloqua forms.
  • New Models Creation – LAL would need to create 1 ticket for every model
  • News Updates – LAL would need to create 1 ticket for per news. If more than 1 news are included in the PPT, 1 subtask would be created for subsequent news articles.
  • Pricelist and Brochures – LAL’s would need to create a ticket for each Brand/market/Platform. First 10 pricelists/brochures would be added, updated or removed on the main task. For the 11th pricelist/brochure, IDC would go ahead and create 1 sub task for the next set of 10 pricelists/brochures.
  • Promotions CPS - For markets where CPS has been implemented, LAL would need to create 1 ticket comprising of all models for each Brand/Market/Language.
  • Promotions Non-CPS - For requests under this sub category LAL are required to create 1 main ticket for each Brand/market/language/Platform and requirements should be attached as PPT. IDC would go ahead and check the number of slides per PPT and accommodate up to 5 slides on each ticket. From the 6st slide, IDC would go ahead and create a sub task for the next 20 set of slides. Please also note, that each slide of a PPT document can have only 1-2 low complexity change.
  • Email and Landing Page category – LAL would need to create 1 ticket per email/landing page for each market/brand/language
    NOTE: Any category/sub category where there is a need to involve graphics team, a sub task would be raised for the activity.

4. Change Management

An editorial change request in the Jira ticketing system happens when the customer requests variations to the initial brief attached to the Jira ticket when opened. JIRA Ticket rule for Change management would be the same for all stages of the ticket.

Banner Category

For all sub categories under this category, ticketing rules would remain the same for change request. When a market requests for alteration or changes within the scope of requests, LAL is expected to attach an updated PPT on the JIRA ticket. IDC will consider the total development activity required within the life span of the ticket and 1 sub task for 12 banner formats
For Ex – If a market creates a ticket having 20 banner formats. IDC team would analyze the request and create 19 sub tasks + 1 main ticket = 20 ticket for the 20 banner formats requested in the ticket. Now, while the ticket is still in progress if LC decides to change the price/legal text to be displayed on the banners. IDC would now have to change the price/legal text on all the 20 banner formats. In such circumstances, IDC would go ahead and create 2 sub task for this change request. 1st sub task would accommodate 12 banners and the 2nd sub task would accommodate 8 banners Every time a new change is requested by the market, a new sub task would be created.

Content Editorial Category

For all sub categories under Content Editorial categories, the ticket rule would remain same for change request. When a market requests for alteration or changes within the scope of requests, LAL is expected to attach an updated PPT on the JIRA ticket. IDC will consider the overall development activity required within the ticket at various stages and based on the sub category create appropriate number of sub tasks.
For Ex – If market has raised a request under “multiple changes” sub category for content editorial activity with 65 changes in a PPT, IDC would create 3 subtasks. Main ticket would consider 25 changes, 1st sub task would account slides 26-50, 2nd sub task would account for slides 51-65. While the ticket is in QC, if market adds on an updated PPT of 15 changes, IDC would accommodate the first 10 changes in the 2rd sub task (since the bandwidth for multiple content changes is 25 changes) and create a 3rd sub task for the remaining 5 changes. While waiting for approval on these 80 (65 first request + 15 change request) changes, if client request for some changes, LAL is expected to add the new PPT on the ticket. IDC will consider the number of changes on the new PPT and create 1 new sub task if the number of changes is within 25 changes. If within these slides, graphical changes are required and IDC needs to involve the graphics team, 1 sub task would be created for graphics activity. The slides requiring graphical changes would be excluded from the count and included in graphics counts. For Graphical work, 1 ticket would accommodate 6 image changes as mentioned in the Change Management rule for Banner Category.
